Graph architecture

The graph keeps routing decisions explicit and sends every path through finalize before END.

intakeNormalize the ticket query and create the first audit event.
classifyUse structured LLM output to choose simple, tool, missing info, risky, or error.
toolRun mock support action or lookup and record tool results.
evaluateSet success or needs_retry to control the bounded loop.
approvalGate risky side effects before the tool path proceeds.
finalizeClose the workflow with a final audit event.

Persistence evidence

Every run is bound to a stable thread id, and the checkpointer can replay history for that thread.

Database configuration

The lab config uses `checkpointer: postgres` and `database_url: env:DATABASE_URL`, so the Postgres connection string stays in `.env`.

  • Thread ids are generated as `thread-<scenario_id>`.
  • `thread_config(thread_id)` binds graph invocations to checkpoint history.
  • `state_history_evidence(graph, thread_id)` summarizes replayable snapshots.

Recovery story

Postgres persistence supports process restart evidence because checkpoint state is stored outside memory.

  • Start the Postgres service from `docker-compose.yml`.
  • Run scenarios with the same `DATABASE_URL` from `.env`.
  • Inspect state history using the original scenario thread id.

Failure analysis

The workflow avoids the two biggest production risks: unsafe side effects and infinite retries.

Retry and dead letter

Tool failures produce `evaluation_result = needs_retry`. The retry route checks `attempt < max_attempts`; exhausted runs go to `dead_letter` and still finalize cleanly.

Risky action gate

Refund, delete, cancellation, and email requests go through `risky_action` and `approval` before any tool side effect can proceed.
