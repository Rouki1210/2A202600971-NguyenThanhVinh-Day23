## 2. Architecture

The workflow is a LangGraph `StateGraph` for support-ticket orchestration. It starts at `intake`, normalizes the query, then calls `classify` to choose one route: `simple`, `tool`, `missing_info`, `risky`, or `error`.

Route behavior:

| Route | Path |
|---|---|
| simple | `intake -> classify -> answer -> finalize -> END` |
| tool | `intake -> classify -> tool -> evaluate -> answer -> finalize -> END` |
| missing_info | `intake -> classify -> clarify -> finalize -> END` |
| risky | `intake -> classify -> risky_action -> approval -> tool -> evaluate -> answer -> finalize -> END` |
| error | `intake -> classify -> retry -> tool -> evaluate -> retry/tool or dead_letter -> finalize -> END` |

The retry route is bounded by `attempt < max_attempts`. When the workflow exhausts retries, it moves to `dead_letter` and still terminates through `finalize`.

## 3. State schema

The state is intentionally small and serializable. Append-only fields use LangGraph reducers so audit history is preserved, while current decision fields are overwritten.

| Field | Reducer | Why |
|---|---|---|
| `thread_id` | overwrite | stable checkpoint identity for each scenario run |
| `scenario_id` | overwrite | links graph output back to the scenario |
| `query` | overwrite | normalized support-ticket text |
| `route` | overwrite | current classified route |
| `risk_level` | overwrite | records whether the request is risky |
| `attempt` | overwrite | current retry attempt count |
| `max_attempts` | overwrite | upper bound for retry loop |
| `evaluation_result` | overwrite | retry gate after tool evaluation |
| `pending_question` | overwrite | clarification question for missing information |
| `proposed_action` | overwrite | risky action awaiting approval |
| `approval` | overwrite | human/mock approval decision |
| `final_answer` | overwrite | final user-facing response |
| `messages` | append | compact trace of node-level messages |
| `tool_results` | append | preserves each tool attempt result |
| `errors` | append | preserves retry/failure history |
| `events` | append | audit trail used by metrics and report generation |

## 6. Failure analysis

1. Retry or tool failure: tool calls can return transient `"ERROR"` results. The `evaluate` node converts that into `evaluation_result = "needs_retry"`, and `route_after_retry` prevents infinite loops by sending exhausted requests to `dead_letter`.

2. Risky action without approval: refund, delete, cancellation, and email actions are routed to `risky_action` first. The graph then requires `approval` before proceeding to the tool path. If approval is rejected, the graph asks for clarification instead of executing the side effect.

3. Missing or vague user input: vague requests such as "Can you fix it?" are routed to `clarify`, which sets `pending_question` and avoids hallucinating an unsupported answer.

## 7. Persistence / recovery evidence

Each scenario starts with a deterministic `thread_id` such as `thread-S01_simple`, and `build_graph()` accepts a checkpointer. The lab config uses `checkpointer: postgres` with `database_url: env:DATABASE_URL`, so the actual database connection string is read from `.env` instead of being hard-coded in YAML. SQLite support is also available through `build_checkpointer("sqlite", database_url)`, which creates a database file and returns `SqliteSaver` when `langgraph-checkpoint-sqlite` is installed.

To demonstrate recovery, start the Postgres service, run scenarios, and inspect or replay state history using the same `thread_id`. For a file-based demo, switch `configs/lab.yaml` to `checkpointer: sqlite` and set `database_url: outputs/checkpoints.sqlite`.
